# Implementation Plan: Deezer Integration via ARL

**Branch**: `007-deezer-integration` | **Date**: 2026-04-24 | **Spec**: [spec.md](spec.md)  
**Input**: Feature specification from `/specs/007-deezer-integration/spec.md`

## Summary

Add Deezer as a second sync target alongside Spotify. Authentication uses the ARL cookie
(stored in `~/.playlist_sync/config.ini` `[DEEZER]` section). Five CLI commands (`import`,
`sync`, `match`, `compare`, `duplicates`) are exposed under a `deezer` group in `main.py`,
mirroring the existing `spotify` group. Track resolution follows a four-step order: cached
`TXXX:DEEZER` URL → SKIP sentinel → ISRC lookup (via public Deezer API) → fuzzy search
(via GW). New modules: `api/deezer.py`, `tracks/deezer_track.py`,
`playlists/deezer_playlist.py`, `matchers/deezer_matcher.py`,
`playlists/deezer_compare.py`. Existing `LocalTrack.embed_match()` and
`LocalTrack.service_ref()` already support Deezer without modification.

## Technical Context

**Language/Version**: Python 3.11+  
**Primary Dependencies**: `deezer-py==1.3.7` (Deezer, GW, API classes), `mutagen`, `music-tag`, `click==8.1.7`, `tqdm==4.66.2`, `spotipy==2.23.0` (existing)  
**Storage**: Local audio file `TXXX:DEEZER` custom tags (MP3/FLAC/M4A via mutagen) — no database  
**Testing**: pytest  
**Target Platform**: Desktop (Windows/macOS/Linux)  
**Project Type**: CLI tool  
**Performance Goals**: Import 100 tracks in < 5 minutes on typical broadband (SC-001)  
**Constraints**: ARL must never appear in any log/output (FR-018); transient errors warn-and-continue (FR-017); `import` always creates a new playlist (FR-019); `mypy` strict + `ruff` clean  
**Scale/Scope**: Single-user, single-machine playlist management

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

- [x] **Principle I (Clean Code)**: All new identifiers are precise and descriptive (`DeezerMatcher`,
  `DeezerTrack`, `DeezerPlaylist`, `get_deezer_client`, `_is_valid_deezer_url`,
  `DeezerAuthenticationError`). Constants (`DEEZER_TRACK_URL_PATTERN`, `DEEZER_URL_PREFIX`)
  defined at module scope. Functions modelled on existing SpotifyMatcher/SpotifyTrack —
  all under ~30 lines. Guard clauses used to keep nesting ≤ 3.
- [x] **Principle II (SOLID)**: `DeezerMatcher` extends `Matcher` (SRP: resolves tracks only);
  `DeezerPlaylist` extends `Playlist + SyncTarget` (SRP: manages one Deezer playlist);
  `DeezerTrack` extends `ServiceTrack` (SRP: represents one Deezer catalog track);
  `get_deezer_client()` is injected into `DeezerMatcher`/`DeezerPlaylist` — never
  instantiated inside business logic. `DeezerMatcher.get_instance()` overrides the base
  class singleton to call `get_deezer_client()` instead of `get_spotify_client()`.
- [x] **Principle III (DRY)**: `LocalTrack.embed_match()` and `_get_custom_tag()` already
  generic — no Deezer-specific tag logic duplicated. `_match_constraints()` and
  `_is_valid_isrc()` patterns defined once per module. URL-validation regex defined once in
  `deezer_matcher.py`.
- [x] **Principle IV (Readability First)**: No premature optimization. Generators used for
  iterating large playlists (consistent with existing tqdm pattern). No speculative caching.
- [x] **Principle V (Unit Tests)**: Every new concrete class will have tests:
  `test_deezer_track.py` (all properties, both GW and API data shapes),
  `test_deezer_matcher.py` (all four resolution steps, SKIP, non-Latin, transient error),
  `test_deezer_playlist.py` (create, load, add, remove). Isolation via fakes/mocks —
  no filesystem or network access in tests.
- [x] **Principle VI (Type Safety)**: All public APIs carry complete type hints. `deezer-py`
  ships no type stubs → `# type: ignore[no-untyped-call]` where needed (same pattern as
  spotipy). Custom exceptions: `DeezerAuthenticationError`. Code will pass `mypy` strict.
- [x] **Quality Gates**: All gates (`ruff format .`, `ruff check .`, `mypy .`, `pytest tests/`)
  must pass before the feature is considered done.
